# Proposal: "Install custom form" button in the Configurator

**Status:** Approved (senior-dev review complete) — ready to implement
**Branch:** `feature/install-custom-form-button`
**Author:** Miquel Adell

## Context

The DHIS2 App Hub rejected the autogenerated-forms submission. The single
structural blocker called out by the reviewer:

> The App Hub cannot distribute the whole app. The configurator zip is only
> half of the deliverable. To actually use the forms, an administrator still
> has to download `custom-data-form.html` from a Github release and paste it
> (or `yarn deploy` with admin credentials) into each target dataset's html
> code. This second step lives outside the App Hub's scope.

(Full review: <https://docs.google.com/document/d/1aonHaY4-si5P5Rc_FV-2qCkyxNauODE9JlC9QNqMH2Q/>)

The other rejection points (cli-app-scripts migration, screenshots, header bar
on v2.41, MUI → @dhis2/ui drift) are out of scope for this branch — see
[Out of scope](#out-of-scope).

## Goal

Make the App Hub `.zip` self-sufficient: an administrator installs the
configurator from the App Hub, opens it, picks a dataset, and clicks **Install
custom form** — the configurator writes `custom-data-form.html` into that
dataset's data-entry form via the DHIS2 API. No GitHub release download, no
manual HTML paste, no `yarn deploy`.

**Non-goals on this branch:** any other resubmission requirement (see
[Out of scope](#out-of-scope)).

## User-facing change

In the configurator's per-dataset view (where the JSON config is edited
today), add an **Install custom form** button next to the existing Save/Delete
actions:

- **First-time install** — writes `custom-data-form.html` into the dataset's
  custom data-entry form.
- **Re-install / update** — if the dataset already has a custom form, the
  configurator shows the bundled-form version vs. the installed-form version
  (extracted from a version marker embedded in the HTML — see
  [Technical approach §3](#3-detect-installed-form-state)) and lets the admin
  overwrite after a confirmation step.

State display: an indicator on the dataset row in the dataset list — *Custom
form installed* / *Not installed* / *Outdated (vA → vB)* — so the admin sees
the state at a glance.

**v1 scope:** single-dataset install only (one button per dataset detail
view). Bulk multi-dataset install and uninstall are deferred.

## Technical approach

### 1. Bundle the form HTML inside the App Hub zip

The built form lives at `dist/custom-data-form.html` (~3.3 MB, single
inlined HTML file). It is produced by
[`src/scripts/build-autogenerated-forms.ts`](../../src/scripts/build-autogenerated-forms.ts)
(yarn task `build-autogenerated-forms`), which runs `react-scripts build`,
extracts the `<body>` of `build/index.html`, and inlines all scripts and
stylesheets via `inline-source`. The output path is configured in
[`src/scripts/common.ts`](../../src/scripts/common.ts) as
`defaultOptions.autogenFormOutput`.

The configurator's build script
(`scripts.build-configurator-folder` in `package.json`) currently produces:

```
build/
├── (react-scripts output for the configurator variant)
├── i18n/
├── icon.png
└── manifest.webapp
```

Two changes:

1. **Ensure `dist/custom-data-form.html` is current** — make
   `build-configurator` depend on `build-autogenerated-forms` (e.g. via a
   `prebuild-configurator` script) so the bundled HTML is rebuilt from the
   same source as the configurator. Alternative: keep `dist/` committed and
   trust contributors to regenerate.
2. **Copy it into `build/`** — extend the `build-configurator-folder`
   `cp` step to include `dist/custom-data-form.html`.

At runtime the configurator reads it as a static asset via
`fetch('./custom-data-form.html')`.

### 2. Install via the DHIS2 API

DHIS2 stores a dataset's custom form on `dataSet.dataEntryForm` (entity
`DataEntryForm` with field `htmlCode`).

**Authority required:** write access to the target `DataSet`. There are no
separate `F_DATAENTRYFORM_*` authorities — dataset write access is sufficient
for both creating and updating `dataEntryForm`.

**Preferred approach — single metadata request** (handles create and update
in one call, both for first-time install and re-install):

```json
POST /api/metadata
{
    "dataSets": [
        {
            "id": "<dataSetId>",
            "dataEntryForm": {
                "id": "<dataEntryFormId>"
            }
        }
    ],
    "dataEntryForms": [
        {
            "id": "<dataEntryFormId>",
            "style": "NORMAL",
            "htmlCode": "<bundled HTML>"
        }
    ]
}
```

For first-time install, generate a new UID for `dataEntryFormId` (via
`GET /api/system/id`). For re-install, read the existing
`dataSet.dataEntryForm.id` and reuse it.

The individual endpoints (`POST /api/dataEntryForms`,
`PUT /api/dataEntryForms/{id}`, `PATCH /api/dataSets/{id}`) remain a valid
fallback if the metadata endpoint proves problematic for any reason.

### 3. Detect installed-form state

The build step that produces `custom-data-form.html` injects a version marker
near the top of the file:

```html
<!-- d2-autogen-forms-custom-form vX.Y.Z -->
```

Where `vX.Y.Z` matches `package.json` `version`. To determine state, the
configurator reads `dataSet.dataEntryForm.htmlCode`, parses the marker, and
compares to the bundled version. Three states:

| State | Condition |
|---|---|
| Not installed | `dataSet.dataEntryForm` is null |
| Installed (current) | marker version === bundled version |
| Outdated | marker version !== bundled version (or marker absent) |

A missing marker (HTML installed manually before this feature existed) is
treated as *Outdated* with a generic "unknown version" label.

## Open questions for validation

The senior-dev review should focus on these:

1. ~~**API shape**~~ — *Resolved.* Use `POST /api/metadata` with both
   `dataSets` and `dataEntryForms` objects in one request (see Technical
   approach §2). Individual endpoints remain a fallback.
2. ~~**Authorities**~~ — *Resolved.* `F_DATAENTRYFORM_ADD/DELETE` do not
   exist. Write access to the `DataSet` is the only requirement.
3. **Confirmation UX for re-install** — overwriting an existing custom form
   is destructive. Plain confirm modal, or modal with diff/preview of what
   will be replaced?
4. **Version-marker injection point** — `dist/custom-data-form.html` is
   produced by [`src/scripts/build-autogenerated-forms.ts`](../../src/scripts/build-autogenerated-forms.ts)
   (yarn task `build-autogenerated-forms`). The marker can be injected into
   the inlined HTML right before `fs.writeFileSync(dest, html3)` near the
   end of `convertReactHtmlToInline()`. *Senior dev: please confirm this is
   the right place and that the source `package.json.version` is the
   appropriate version to embed (vs. `appVersions.configurator`).*

## Risks

- **Authority gaps in production** — admins running the configurator may not
  have write access to the target `DataSet`. Mitigation: the configurator
  detects insufficient access and disables the install button with an
  explanatory tooltip; the README documents the required access level.
- **Existing custom form managed by another app or hand-written** — a dataset
  may have a `dataEntryForm` not produced by this configurator (no version
  marker). Mitigation: detection treats *missing marker* as *Outdated /
  unknown* and the confirmation modal explicitly warns that the existing
  form will be replaced. No silent overwrites.
- **Marker collisions** — another tool could happen to embed an HTML comment
  with a similar pattern. Mitigation: the marker uses a project-specific
  prefix (`d2-autogen-forms-custom-form`) and is parsed strictly.
- **App Hub zip size** — `custom-data-form.html` is ~3.3 MB. Adding it
  pushes the zip past the typical configurator-app size. Mitigation: confirm
  App Hub's max upload size; the previous distribution model already shipped
  the same artifact (just separately), so total bytes administrators handle
  are unchanged.
- **API differences between v2.41 and v2.42** — `dataEntryForm` field
  semantics may have shifted across versions. Mitigation: smoke-test on both
  before opening the resubmission PR.
- **Resubmission still rejected on secondary grounds** — even if this
  unblocks the structural complaint, the App Hub may still reject for
  missing screenshots / cli-app-scripts / header bar (all out-of-scope here).
  Mitigation: track each as a separate branch/issue and bundle them into the
  resubmission package.

## Implementation milestones

Suggested order for the implementer; each step is independently shippable
behind feature flags or as separate PRs if preferred:

1. **Bundle the form** — modify `scripts.build-configurator-folder` in
   `package.json` to copy `dist/custom-data-form.html` into `build/`. Add a
   `prebuild-configurator` (or chain into `build-configurator`) so
   `build-autogenerated-forms` runs first and `dist/` is fresh. Verify
   `d2-autogen-configurator.zip` contains the form HTML. *Ships nothing
   user-visible — just makes the zip self-sufficient.*
2. **Inject version marker** — extend
   [`src/scripts/build-autogenerated-forms.ts`](../../src/scripts/build-autogenerated-forms.ts)
   (right before the final `fs.writeFileSync`) to prepend
   `<!-- d2-autogen-forms-custom-form vX.Y.Z -->`, reading `vX.Y.Z` from
   `package.json.version` at build time.
3. **Read installed-form state** — domain/use-case to fetch
   `dataSet.dataEntryForm.htmlCode`, parse the marker, and return state
   (Not installed / Installed (current) / Outdated). Show a status pill on
   the dataset list.
4. **Install action (no existing form)** — wire the **Install custom form**
   button to `POST /api/metadata` (generate a new UID via
   `GET /api/system/id`, then send both `dataSets` and `dataEntryForms`
   objects in one request). Block on confirmation modal.
5. **Re-install action (existing form)** — extend (4) to reuse the existing
   `dataEntryForm.id` in the same `POST /api/metadata` payload. Confirmation
   modal warns about overwrite.
6. **Rebrand bits** — `package.json` description, README framing, App Hub
   description. (Can land at any point; lowest risk.)
7. **Smoke tests** — manual on v2.41 + v2.42/Data Entry app v102+.
